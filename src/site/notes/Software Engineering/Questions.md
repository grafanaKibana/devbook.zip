<p dir="auto">Mapping is the transformation of data from one shape/type to another (for example, Domain Entity -&gt; DTO -&gt; API response model).<br>

<li dir="auto">code generation / source generators for mappings<br>

</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What are serialization and deserialization?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Serialization converts an in-memory object graph into a format that can be stored or transmitted (for example, JSON text or a binary payload).<br>
Deserialization is the reverse process: converting that stored/transmitted representation back into objects.<br>
Common uses: API payloads, persistence, caching, messaging.<br>

<p dir="auto">JSON (JavaScript Object Notation) is a lightweight text data format based on objects (name/value pairs) and arrays.<br>

In .NET, JSON is commonly handled with <code>System.Text.Json</code> (built-in) or Newtonsoft.Json.<br>

<p dir="auto">A request is received by the host (for example, Kestrel) and then flows through an ordered chain of middleware. Middleware can add features (routing, authN/authZ, CORS, compression, etc.), select an endpoint, and finally execute the endpoint (MVC action, Minimal API handler, etc.). On the way back out, the middleware chain unwinds, allowing post-processing of the response.<br>

</div></div></div><h3>CSharp</h3><h4>Fundamentals</h4><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">How is asynchrony different from multithreading?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Asynchrony is about not blocking while waiting (especially for I/O). An <code>async</code> method can release the current thread while awaiting, and continue later.<br>
Multithreading is about executing work on multiple threads concurrently (for example, for parallel CPU-bound work). Async code can be single-threaded and still be asynchronous.<br>
<em><a class="internal-link" data-href="Software Engineering/01 Programming/NET/CSharp/Fundamentals/Async Await.md" href="Software Engineering/01 Programming/NET/CSharp/Fundamentals/Async Await.md" target="_blank" rel="noopener nofollow">Async Await</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is the difference between <code>Thread</code> and <code>Task</code>?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto"><code>Thread</code> is an OS thread you manage directly (heavier, dedicated execution).<br>
<code>Task</code> is a higher-level abstraction representing an asynchronous operation or a unit of work, typically scheduled on the thread pool (and for I/O it may not require a dedicated thread while waiting).<br>

<p dir="auto"><strong><code>string</code></strong> and <strong><code>StringBuilder</code></strong> are two types used for working with strings in C#, but they differ significantly in how they store and manipulate string data, which makes them suitable for different scenarios.</p>
<ol>
<li dir="auto"><strong>Immutability vs mutability</strong>:
<ul>
<li dir="auto"><strong><code>string</code></strong> is immutable. This means that after a string is created, its content cannot be changed. Any modification creates a new string instance in memory.</li>
<li dir="auto"><strong><code>StringBuilder</code></strong> is mutable and is designed for efficient string construction and modification. You can append, modify, and delete characters or substrings inside <strong><code>StringBuilder</code></strong> without creating new string instances.</li>
</ul>
</li>
<li dir="auto"><strong>Performance</strong>:
<ul>
<li dir="auto">Using <strong><code>string</code></strong> for many modifications can lead to creating many temporary objects in memory, which can hurt performance and memory usage. <strong><code>StringBuilder</code></strong> is designed to minimize these temporary allocations and provide better performance for repeated changes.</li>
</ul>
</li>
<li dir="auto"><strong>When to use</strong>:
<ul>
<li dir="auto"><strong><code>string</code></strong> is suitable for small strings or situations where the string does not change (or changes rarely).</li>
<li dir="auto"><strong><code>StringBuilder</code></strong> is better when you need to build strings with many appends/changes, for example when generating long SQL queries, XML documents, JSON strings, and so on.</li>
</ul>
</li>
</ol>
<p dir="auto">Use it when you need to repeatedly modify or build strings inside loops or in scenarios where performance and memory efficiency matter. This is especially useful when working with large amounts of data, such as generating long text output, SQL queries, or XML/JSON structures.</p>
<p dir="auto">When strings are effectively static and change rarely, <strong><code>StringBuilder</code></strong> may be unnecessary, and <strong><code>string</code></strong> is usually the better choice.<br>

Name examples of .NET structs.</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Classes are reference types; structs are value types.<br>
Structs are copied by value, cannot participate in class inheritance, and are often used for small value-like data (e.g., <code>int</code>, <code>DateTime</code>, <code>Guid</code>).</p>
<p dir="auto">Differences:</p>
<ol>
<li dir="auto"><strong>Value:</strong>
<ol>
<li dir="auto">A class variable stores a reference to an object.</li>
<li dir="auto">A structure variable stores the data itself.</li>
</ol>
</li>
<li dir="auto"><strong>Transfer semantics:</strong>
<ol>
<li dir="auto"><strong>Classes</strong> are reference types. When an instance of a class is passed to a method, a reference to the object is passed to the method, and changes to the object will be visible wherever the reference is used.</li>
<li dir="auto"><strong>Structures</strong> are meaningful types. When an instance of a structure is passed to a method, a copy of the value is passed to the method, and changes to the copy do not affect the original object.</li>
</ol>
</li>
<li dir="auto"><strong>In-memory usage:</strong>
<ol>
<li dir="auto"><strong>Classes</strong>, more specifically class object variables are stored in the managed stack, a variable in turn stores not the object itself but a reference to the object in the managed heap.</li>
<li dir="auto"><strong>Structures</strong> are stored in the stack by default, but can also be stored in the managed heap if necessary, for example, when the structure is a class field, or was explicitly packed into an object.</li>
</ol>
</li>
<li dir="auto"><strong>Inheritance:</strong>
<ol>
<li dir="auto"><strong>Classes</strong> support inheritance, which means that one class can be inherited from another, providing the ability to create a hierarchy of classes.</li>
<li dir="auto"><strong>Structures</strong> do not support inheritance. They cannot be inherited from other types and cannot be base classes. Therefore, all structures are <code>sealed</code> by default</li>
</ol>
</li>
<li dir="auto"><strong>Default initialization:</strong>
<ol>
<li dir="auto"><strong>Classes</strong> are initialized by default with the value <strong><code>null</code></strong>.</li>
<li dir="auto"><strong>Structures</strong> are initialized by default with a value corresponding to their data type (e.g. numbers are initialized with null).</li>
</ol>
</li>
<li dir="auto"><strong>Differences in minor application details:</strong>
<ol>
<li dir="auto">Structures cannot have a default constructor(no parameters) or destructor in them</li>
<li dir="auto">Structures can be initialized without using the <code>new</code> operator</li>
<li dir="auto">Structures cannot have abstract or sealed modifiers, as well as members of structures cannot be <code>protected</code> or <code>protected internal</code>, which is logical because structures do not support inheritance.</li>
<li dir="auto">Methods of structures cannot be abstract or virtual for the same reason of absence of inheritance in structures. Also, methods in structures cannot override any methods except those in <code>System.ValueType</code>.</li>
<li dir="auto">Structures cannot contain fields whose size is not yet known (including those of their own type).</li>
</ol>
</li>
<li dir="auto">Examples:
<ol>
<li dir="auto"><code>Struct</code>: <strong><code>System.String</code>, <code>System.IO.File</code></strong></li>
<li dir="auto"><code>Class</code>: <strong><code>System.Int32</code> (and other numeric data types), <code>System.DateTime</code>, <code>System.Drawing.Point</code></strong></li>
</ol>
</li>
</ol>

<p dir="auto">By default, the <strong><code>HashCode</code></strong> method uses an internal algorithm that is based on the address of the object in memory. This algorithm is based on hash code <strong>combining</strong>, which provides a more efficient and uniform distribution of hash codes.</p>
<p dir="auto">Briefly, the algorithm can be described as follows:</p>
<ol>
<li dir="auto">Hash<strong>initialization</strong>: The hash code is initialized with some initial value (usually a prime number, such as 17 or 23).</li>
<li dir="auto"><strong>Combining</strong> hashes: The hash code of an object is computed by combining the hashes of its constituent parts. For example, if an object contains multiple fields, the hashes of these fields are combined into one common hash code.</li>
<li dir="auto"><strong>Multiplication by a prime number</strong>: The resulting hash code is multiplied by some prime number (e.g., 31 or 59).</li>
<li dir="auto"><strong>Combining with new</strong> data: When new data is added (e.g., when combining hashes of additional objects), their hashes are also combined with the existing hash code of the object.</li>
<li dir="auto"><strong>Return hash</strong>: The resulting hash code is returned as the result of the <strong><code>HashCode</code></strong> method.</li>
</ol>
<p dir="auto">However, this algorithm can be overridden in derived classes to better manage object hash codes based on their contents.<br>
Two objects that are equal should return identical hash codes.<br>
However, the reverse is not true: identical hash codes do not imply object equality, since different (unequal) objects can have identical hash codes.<br>

<em><a class="internal-link" data-href="Software Engineering/01 Programming/NET/Runtime/Memory Leaks.md" href="Software Engineering/01 Programming/NET/Runtime/Memory Leaks.md" target="_blank" rel="noopener nofollow">Memory Leaks</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is the disposable (dispose) pattern?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">A standard way to implement <code>IDisposable</code> so both explicit cleanup (<code>Dispose()</code>) and (optionally) finalization are supported.<br>
Typical shape: <code>Dispose()</code> calls <code>Dispose(true)</code> and then <code>GC.SuppressFinalize(this)</code>; a finalizer (if needed) calls <code>Dispose(false)</code>; <code>Dispose(bool disposing)</code> releases unmanaged resources and, when <code>disposing</code> is true, also disposes managed fields.<br>
<em><a class="internal-link" data-href="Software Engineering/01 Programming/NET/Runtime/Memory Leaks.md" href="Software Engineering/01 Programming/NET/Runtime/Memory Leaks.md" target="_blank" rel="noopener nofollow">Memory Leaks</a></em></p>

<p dir="auto">Insertion sort grows a sorted prefix by inserting each next element into its correct position. It is fast for small inputs and nearly-sorted data, and it is a common building block inside hybrid sorts.<br>

</div></div></div><h1 id="03-data-persistance">03 Data Persistance</h1><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is a database?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">A database is an organized collection of data stored in a structured way so it can be efficiently created, read, updated, and deleted. In practice you interact with it through a DBMS (Database Management System), which provides storage, indexing, querying, transactions, security, backups, etc.<br>
<em><a class="internal-link" data-href="Software Engineering/03 Data Persistance/03 Data Persistance.md" href="Software Engineering/03 Data Persistance/03 Data Persistance.md" target="_blank" rel="noopener nofollow">03 Data Persistance</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What are the main types of databases?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Common categories are: relational (SQL tables), key-value, document, wide-column, graph, time-series, and search engines. They differ in data model, query capabilities, consistency guarantees, and scaling approach.<br>
<em><a class="internal-link" data-href="Software Engineering/03 Data Persistance/03 Data Persistance.md" href="Software Engineering/03 Data Persistance/03 Data Persistance.md" target="_blank" rel="noopener nofollow">03 Data Persistance</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">Explain ACID</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">ACID describes key transaction guarantees:</p>
<ul>
<li dir="auto">Atomicity: all operations in a transaction succeed or none do.</li>
<li dir="auto">Consistency: committed transactions move the database between valid states (respecting constraints).</li>
<li dir="auto">Isolation: concurrent transactions do not observe each other's intermediate states (controlled by isolation levels).</li>
<li dir="auto">Durability: once committed, data survives crashes (via logging, checkpoints, replication, etc.).<br>
<em><a class="internal-link" data-href="Software Engineering/03 Data Persistance/ACID.md" href="Software Engineering/03 Data Persistance/ACID.md" target="_blank" rel="noopener nofollow">ACID</a></em></li>
</ul>
</div></div></div><h2 id="03-data-persistance-nosql">NoSQL</h2><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What are non-relational databases?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Non-relational databases (often grouped under the term NoSQL) store and query data without the classic relational model of normalized tables and joins. They typically favor flexible schemas and horizontal scaling, and come in several major families: key-value, document, wide-column, and graph databases.<br>
<em><a class="internal-link" data-href="Software Engineering/03 Data Persistance/NoSQL/NoSQL.md" href="Software Engineering/03 Data Persistance/NoSQL/NoSQL.md" target="_blank" rel="noopener nofollow">NoSQL</a></em></p>
</div></div></div><h2 id="03-data-persistance-orms">ORMs</h2><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">How to choose Code First vs Database First?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Use Code First when you want the code model to be the source of truth (new projects, strong domain model, migrations as part of delivery). Use Database First when you already have a database schema you must integrate with (legacy DB, shared DB, strict DBA-controlled schema) and want to generate the model from it.<br>
<em><a class="internal-link" data-href="Software Engineering/03 Data Persistance/ORMs/Entity Framework.md" href="Software Engineering/03 Data Persistance/ORMs/Entity Framework.md" target="_blank" rel="noopener nofollow">Entity Framework</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What inheritance mapping strategies exist in Entity Framework?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">The common strategies are: TPH (Table Per Hierarchy, single table with a discriminator), TPT (Table Per Type, separate table per type), and TPC (Table Per Concrete type, separate table per concrete type). They trade off query simplicity, normalization, and performance.<br>
<em><a class="internal-link" data-href="Software Engineering/03 Data Persistance/ORMs/Entity Framework.md" href="Software Engineering/03 Data Persistance/ORMs/Entity Framework.md" target="_blank" rel="noopener nofollow">Entity Framework</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">How do you do a backward-compatible schema migration with zero downtime (expand-contract, backfill, dual reads/writes), and what can go wrong?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Use an expand-contract migration:<br>
add new schema elements in a backward-compatible way (nullable columns/tables/indexes), deploy code that writes to both old and new (or writes new while keeping old readable), then backfill in batches.<br>
Switch reads behind a feature flag when data is consistent, keep a fallback path temporarily, and only then contract (remove old columns) once all writers/readers are updated.<br>
Failure modes: wrong deploy order, backfill racing with live writes, long-running migrations/locks, missing defaults/null handling, read-model inconsistency, and hard rollbacks after data has diverged.<br>
<em><a class="internal-link" data-href="Software Engineering/03 Data Persistance/ORMs/Entity Framework.md" href="Software Engineering/03 Data Persistance/ORMs/Entity Framework.md" target="_blank" rel="noopener nofollow">Entity Framework</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is an ORM?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">An ORM (Object-Relational Mapper) maps relational database tables/rows to objects and translates queries/changes in code into SQL. It helps with CRUD, tracking changes, relationships, and migrations, at the cost of an abstraction layer you must understand to avoid performance issues.<br>

</div></div></div><h2 id="03-data-persistance-sql">SQL</h2><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">Why can't a table have two clustered indexes?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Reference: <a data-tooltip-position="top" aria-label="https://habr.com/ru/post/247373/#01" rel="noopener nofollow" class="external-link" href="https://habr.com/ru/post/247373/#01" target="_blank">Article</a><br>
A clustered index is effectively the table. When you create a clustered index, the storage engine orders the table's rows according to the index key (ascending or descending). A clustered index is not a separate structure like other indexes; it is the way the table data itself is organized to enable efficient access to rows.</p>
<p dir="auto">Imagine you have a table that stores a sales history. The Sales table contains information such as order id, line id, product id, quantity, order number and date, and so on. You create a clustered index on the <em>OrderID</em> and <em>LineID</em> columns, sorted ascending, as shown in the following T-SQL:</p>
<pre class="language-sql"><code class="language-sql is-loaded"><span class="token keyword">CREATE</span> <span class="token keyword">UNIQUE</span> <span class="token keyword">CLUSTERED</span> <span class="token keyword">INDEX</span> ix_oriderid_lineid
<span class="token keyword">ON</span> dbo<span class="token punctuation">.</span>Sales<span class="token punctuation">(</span>OrderID<span class="token punctuation">,</span> LineID<span class="token punctuation">)</span><span class="token punctuation">;</span>
</code><button class="copy-code-button"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-copy"><rect x="8" y="8" width="14" height="14" rx="2" ry="2"></rect><path d="M4 16c-1.1 0-2-.9-2-2V4c0-1.1.9-2 2-2h10c1.1 0 2 .9 2 2"></path></svg></button></pre>
<p dir="auto">When you run this script, all rows in the table are physically ordered first by OrderID and then by LineID, but the data still remains a single logical structure: the table. For this reason, you cannot create two clustered indexes. There is only one table with one set of data, and that data can only be ordered one way at a time.<br>

Functions usually return a value (scalar/table) and are often more restricted in side effects (varies by DB engine).<br>

</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is a trigger?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">A trigger is database code that automatically runs in response to events (e.g., <code>INSERT</code>, <code>UPDATE</code>, <code>DELETE</code>, sometimes DDL events). It is commonly used for enforcing rules, auditing, and maintaining derived data.<br>
<em><a class="internal-link" data-href="Software Engineering/03 Data Persistance/SQL/SQL.md" href="Software Engineering/03 Data Persistance/SQL/SQL.md" target="_blank" rel="noopener nofollow">SQL</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is a subquery?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">A subquery is a query nested inside another query (in <code>SELECT</code>, <code>FROM</code>, or <code>WHERE</code>/<code>HAVING</code>). It can be correlated (depends on outer row) or non-correlated, and can be used for filtering, projection, or building derived tables.<br>
<em><a class="internal-link" data-href="Software Engineering/03 Data Persistance/SQL/SQL.md" href="Software Engineering/03 Data Persistance/SQL/SQL.md" target="_blank" rel="noopener nofollow">SQL</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is a transaction?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">A transaction groups multiple operations into a single unit of work that either fully commits or rolls back. Transactions provide guarantees like atomicity and isolation, controlled by the database's transaction and isolation model.<br>
<em><a class="internal-link" data-href="Software Engineering/03 Data Persistance/SQL/SQL.md" href="Software Engineering/03 Data Persistance/SQL/SQL.md" target="_blank" rel="noopener nofollow">SQL</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is a Common Table Expression?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">A CTE (Common Table Expression) is a named, temporary result set defined with <code>WITH</code> and used within a single statement. It improves readability, enables query composition, and supports recursion for hierarchical data.<br>
<em><a class="internal-link" data-href="Software Engineering/03 Data Persistance/SQL/SQL.md" href="Software Engineering/03 Data Persistance/SQL/SQL.md" target="_blank" rel="noopener nofollow">SQL</a></em></p>
</div></div></div><h1 id="04-networks">04 Networks</h1><h2 id="04-networks-protocols">Protocols</h2><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is the difference between HTTP and HTTPS?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">HTTPS is HTTP over TLS.<br>
It provides confidentiality (encryption), integrity (tamper detection), and server authentication via certificates.<br>
Plain HTTP has no built-in protection against eavesdropping or manipulation.<br>
<em><a class="internal-link" data-href="Software Engineering/04 Networks/Protocols/HTTP &amp; HTTPS.md" href="Software Engineering/04 Networks/Protocols/HTTP &amp; HTTPS.md" target="_blank" rel="noopener nofollow">HTTP &amp; HTTPS</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What HTTP methods exist?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Common methods: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS.<br>
Less common: TRACE, CONNECT.<br>
<em><a class="internal-link" data-href="Software Engineering/04 Networks/Protocols/HTTP &amp; HTTPS.md" href="Software Engineering/04 Networks/Protocols/HTTP &amp; HTTPS.md" target="_blank" rel="noopener nofollow">HTTP &amp; HTTPS</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is the difference between GET and POST?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">GET is intended for retrieving a resource representation; it is safe and can be cached.<br>
POST is intended for submitting data to be processed (often creating a new resource or triggering an action); it is not safe and is generally not idempotent.<br>
GET parameters are typically in the URL query; POST usually sends a request body.<br>
<em><a class="internal-link" data-href="Software Engineering/04 Networks/Protocols/HTTP &amp; HTTPS.md" href="Software Engineering/04 Networks/Protocols/HTTP &amp; HTTPS.md" target="_blank" rel="noopener nofollow">HTTP &amp; HTTPS</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What do idempotency and safety mean for HTTP methods?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Safe means the request does not change server state (for example, GET, HEAD, OPTIONS).<br>
Idempotent means repeating the same request has the same effect as doing it once (for example, GET, PUT, DELETE).<br>
A method can be idempotent but not safe (for example, DELETE).<br>
<em><a class="internal-link" data-href="Software Engineering/04 Networks/Protocols/HTTP &amp; HTTPS.md" href="Software Engineering/04 Networks/Protocols/HTTP &amp; HTTPS.md" target="_blank" rel="noopener nofollow">HTTP &amp; HTTPS</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">URI vs URL: what is the difference?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">URI (Uniform Resource Identifier) identifies a resource.<br>
URL (Uniform Resource Locator) is a type of URI that also tells you how and where to access the resource (scheme plus location).<br>
Example: <code>https://example.com/a/b?x=1#top</code> is a URL (and therefore a URI).<br>
<em><a class="internal-link" data-href="Software Engineering/04 Networks/Protocols/Protocols.md" href="Software Engineering/04 Networks/Protocols/Protocols.md" target="_blank" rel="noopener nofollow">Protocols</a></em></p>
</div></div></div><h2 id="04-networks-transport-sockets">Transport &amp; Sockets</h2><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">UDP vs TCP: what is the difference?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">TCP is connection-oriented and provides reliable, ordered delivery with retransmissions, flow control, and congestion control.<br>
UDP is connectionless and sends independent datagrams with no built-in reliability or ordering.<br>
Typical uses: TCP for web traffic and most application protocols; UDP for real-time media, DNS, and protocols that implement reliability themselves.<br>
<em><a class="internal-link" data-href="Software Engineering/04 Networks/Transport &amp; Sockets/TCP IP.md" href="Software Engineering/04 Networks/Transport &amp; Sockets/TCP IP.md" target="_blank" rel="noopener nofollow">TCP IP</a></em></p>
</div></div></div><h1 id="05-architecture">05 Architecture</h1><h2 id="05-architecture-application-architecture">Application Architecture</h2><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is multi-layered architecture?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Multi-layered architecture splits the system into layers such as Presentation, Application (use cases), Domain (business rules), and Infrastructure/Data access. Each layer has a focused responsibility and communicates through well-defined interfaces, which improves maintainability and testability.<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Application Architecture/Layered Architecture.md" href="Software Engineering/05 Architecture/Application Architecture/Layered Architecture.md" target="_blank" rel="noopener nofollow">Layered Architecture</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is Onion Architecture?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Onion Architecture is a layered style where the Domain is at the center and dependencies point inward. Outer layers (infrastructure, UI, frameworks) depend on inner layers; inner layers do not depend on details. This is usually enforced by defining abstractions (interfaces) in the inner layers and implementing them in the outer layers.<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Application Architecture/Layered Architecture.md" href="Software Engineering/05 Architecture/Application Architecture/Layered Architecture.md" target="_blank" rel="noopener nofollow">Layered Architecture</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is MVC and why is it used?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">MVC stands for Model-View-Controller. The Model represents the domain/data and business rules, the View is the UI (rendering), and the Controller handles incoming input/requests, coordinates work, and selects the response/view. The separation helps keep UI concerns out of business logic and makes the system easier to test and evolve.<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Application Architecture/MVC MVVM.md" href="Software Engineering/05 Architecture/Application Architecture/MVC MVVM.md" target="_blank" rel="noopener nofollow">MVC MVVM</a></em></p>
</div></div></div><h2 id="05-architecture-architecture-styles">Architecture Styles</h2><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">Microservices vs monolith: what is the difference and when should you choose which?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">A monolith is a single deployable unit (often with one codebase and one process). Microservices split the system into multiple deployable services with their own boundaries and often their own data.</p>
<p dir="auto">Typical trade-offs:</p>
<ul>
<li dir="auto">Microservices: independent deployments and scaling, but higher operational complexity (service discovery, observability, <a data-tooltip-position="top" aria-label="Distributed Transactions" data-href="Distributed Transactions" href="Distributed Transactions" class="internal-link" target="_blank" rel="noopener nofollow">distributed transactions</a>, versioning, testing).</li>
<li dir="auto">Monolith: simpler operations and consistency, but can become hard to evolve if it is not modular.</li>
</ul>
<p dir="auto">A common path is to start with a modular monolith and extract services when there is a clear boundary and a real need.<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Architecture Styles/Microservices.md" href="Software Engineering/05 Architecture/Architecture Styles/Microservices.md" target="_blank" rel="noopener nofollow">Microservices</a></em></p>
</div></div></div><h2 id="05-architecture-distributed-systems">Distributed Systems</h2><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What does REST mean?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">REST (Representational State Transfer) is an architectural style for <a data-tooltip-position="top" aria-label="Distributed Systems" data-href="Distributed Systems" href="Distributed Systems" class="internal-link" target="_blank" rel="noopener nofollow">distributed systems</a>.<br>
Key ideas: resources identified by URIs, stateless requests, cacheable responses, a uniform interface, layered system, and optional code-on-demand.<br>
In practice, RESTful APIs model domain objects as resources and use HTTP methods with consistent semantics.<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Distributed Systems/REST.md" href="Software Engineering/05 Architecture/Distributed Systems/REST.md" target="_blank" rel="noopener nofollow">REST</a></em></p>
</div></div></div><h3>Message Queues</h3><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is Kafka?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Apache Kafka is a distributed event streaming platform built around an append-only log (topics/partitions) and consumer groups.<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Distributed Systems/Message Queues/Kafka.md" href="Software Engineering/05 Architecture/Distributed Systems/Message Queues/Kafka.md" target="_blank" rel="noopener nofollow">Kafka</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">When should MSMQ be used today?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Mostly in legacy/on-prem Windows environments where MSMQ is already part of the system. For new systems, prefer actively supported managed brokers/queues appropriate for your platform and operational model.<br>

</div></div></div><h2 id="05-architecture-patterns">Patterns</h2><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is CQRS?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">CQRS (Command Query Responsibility Segregation) separates the model used for writes (commands that change state) from the model used for reads (queries that return data). This can simplify complex domains and enable different scaling/optimization strategies for reads vs writes. It adds complexity (more moving parts, eventual consistency when using separate read stores), so it is usually applied where the benefits justify the cost.<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Patterns/CQRS.md" href="Software Engineering/05 Architecture/Patterns/CQRS.md" target="_blank" rel="noopener nofollow">CQRS</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is the difference between <code>services.AddTransient</code>, <code>services.AddScoped</code>, and <code>services.AddSingleton</code> in ASP.NET Core DI?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">These methods define the service lifetime:</p>
<ul>
<li dir="auto"><code>Transient</code>: a new instance is created every time the service is requested.</li>
<li dir="auto"><code>Scoped</code>: one instance is created per scope (in web apps, typically per HTTP request).</li>
<li dir="auto"><code>Singleton</code>: one instance is created for the entire application lifetime (the root container).</li>
</ul>
<p dir="auto">Common pitfalls: singletons must be thread-safe; do not inject scoped services into singletons (it effectively becomes a captive dependency); <code>DbContext</code> is typically registered as scoped.</p>

<p dir="auto">Design patterns are proven, repeatable approaches to common design problems (not copy-paste code). They help you communicate intent, reduce accidental complexity, and improve maintainability by encouraging good separation of concerns and loose coupling.<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Patterns/Design Patterns.md" href="Software Engineering/05 Architecture/Patterns/Design Patterns.md" target="_blank" rel="noopener nofollow">Design Patterns</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What categories of patterns exist?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">For GoF design patterns, the classic categories are:</p>
<ul>
<li dir="auto">Creational: object creation (how instances are constructed)</li>
<li dir="auto">Structural: object composition (how classes/objects are arranged)</li>
<li dir="auto">Behavioral: object interaction (how responsibilities and communication are organized)<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Patterns/Design Patterns.md" href="Software Engineering/05 Architecture/Patterns/Design Patterns.md" target="_blank" rel="noopener nofollow">Design Patterns</a></em></li>
</ul>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is an anti-pattern?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">An anti-pattern is a common, recurring solution to a problem that looks reasonable at first but leads to negative consequences (for example, high coupling, poor testability, or performance issues).<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Patterns/Design Patterns.md" href="Software Engineering/05 Architecture/Patterns/Design Patterns.md" target="_blank" rel="noopener nofollow">Design Patterns</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">Name a few patterns from each category and the basic idea behind them.</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Creational:</p>
<ul>
<li dir="auto">Factory Method: delegate creation to subclasses / factories to decouple callers from concrete types.</li>
<li dir="auto">Builder: construct complex objects step-by-step, separating construction from representation.</li>
</ul>
<p dir="auto">Structural:</p>
<ul>
<li dir="auto">Adapter: make incompatible interfaces work together.</li>
<li dir="auto">Decorator: add behavior by wrapping objects instead of subclassing.</li>
</ul>
<p dir="auto">Behavioral:</p>
<ul>
<li dir="auto">Strategy: swap algorithms behind a common interface.</li>
<li dir="auto">Observer: publish/subscribe notifications to keep components loosely coupled.<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Patterns/Design Patterns.md" href="Software Engineering/05 Architecture/Patterns/Design Patterns.md" target="_blank" rel="noopener nofollow">Design Patterns</a></em></li>
</ul>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is DDD (Domain-Driven Design)?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">DDD is a set of principles and patterns for building software around a rich domain model. It emphasizes:</p>
<ul>
<li dir="auto">Ubiquitous Language shared by developers and domain experts</li>
<li dir="auto">Bounded Contexts to define clear domain boundaries</li>
<li dir="auto">Tactical patterns like Entities, Value Objects, Aggregates, Repositories, Domain Events</li>
</ul>
<p dir="auto">The goal is to reduce the gap between the business and the code, especially in complex domains.<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Patterns/Domain-Driven Development.md" href="Software Engineering/05 Architecture/Patterns/Domain-Driven Development.md" target="_blank" rel="noopener nofollow">Domain-Driven Development</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is GRASP?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">GRASP (General Responsibility Assignment Software Patterns) is a set of guidelines for assigning responsibilities to classes/objects.</p>
<p dir="auto">Common GRASP principles include: Information Expert, Creator, Controller, Low Coupling, High Cohesion, Polymorphism, Pure Fabrication, Indirection, and Protected Variations.<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Patterns/GRASP.md" href="Software Engineering/05 Architecture/Patterns/GRASP.md" target="_blank" rel="noopener nofollow">GRASP</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What are the Unit of Work and Repository patterns, and when are they needed?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">A Repository provides a collection-like interface for working with aggregates/entities (querying and saving) while hiding persistence details. A Unit of Work (UoW) tracks changes and commits them as a single atomic operation (transaction).</p>
<p dir="auto">They are most useful when you need to coordinate multiple changes that must succeed or fail together, or when you want to decouple the domain/application layers from a specific data access technology.</p>
<p dir="auto">Note: in EF Core, <code>DbContext</code> already acts as both a repository-like gateway and a unit of work (change tracking + <code>SaveChanges</code>).<br>
<em><a class="internal-link" data-href="Software Engineering/05 Architecture/Patterns/Repository &amp; UoW.md" href="Software Engineering/05 Architecture/Patterns/Repository &amp; UoW.md" target="_blank" rel="noopener nofollow">Repository &amp; UoW</a></em></p>
</div></div></div><h1 id="06-development-practices">06 Development Practices</h1><h2 id="06-development-practices-paradigms">Paradigms</h2><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is Event-driven?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Event-driven development builds systems around events (facts that something happened) and reactions to those events.<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Paradigms/Event-driven.md" href="Software Engineering/06 Development Practices/Paradigms/Event-driven.md" target="_blank" rel="noopener nofollow">Event-driven</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is the difference between unit tests and integration tests?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<ul>
<li dir="auto">Scope: unit tests cover a small unit (method/class); integration tests cover interactions between multiple components.</li>
<li dir="auto">Dependencies: unit tests replace external dependencies (DB, filesystem, HTTP) with test doubles; integration tests use real dependencies (or realistic test instances/containers).</li>
<li dir="auto">Speed and determinism: unit tests are typically fast and stable; integration tests are slower and can be more brittle due to environment and I/O.</li>
<li dir="auto">Purpose: unit tests validate logic and design boundaries; integration tests validate wiring, configuration, contracts, and real behavior across boundaries.<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Paradigms/Integration Testing.md" href="Software Engineering/06 Development Practices/Paradigms/Integration Testing.md" target="_blank" rel="noopener nofollow">Integration Testing</a></em></li>
</ul>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What does "object-oriented programming" mean?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">OOP is a programming paradigm where a system is designed as a set of interacting objects that encapsulate state and expose behavior via methods.<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Paradigms/OOP.md" href="Software Engineering/06 Development Practices/Paradigms/OOP.md" target="_blank" rel="noopener nofollow">OOP</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">Name the OOP principles and explain each.</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Commonly: </p>
<ul>
<li dir="auto">Encapsulation: hide internal state and expose a stable API; enforce invariants inside the object.</li>
<li dir="auto">Abstraction: model only essential aspects of a concept and hide irrelevant details.</li>
<li dir="auto">Inheritance: reuse/extend behavior by deriving from a base type (use carefully; composition is often safer).</li>
<li dir="auto">Polymorphism: treat different implementations through a common contract (interface/base type) and get behavior based on the runtime type.<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Paradigms/OOP.md" href="Software Engineering/06 Development Practices/Paradigms/OOP.md" target="_blank" rel="noopener nofollow">OOP</a></em></li>
</ul>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is TDD?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">TDD (Test-Driven Development) is a development practice where you write a test before the production code.<br>
The typical loop is Red Green Refactor:</p>
<ul>
<li dir="auto">Red: write a small test that fails</li>
<li dir="auto">Green: implement the simplest code to make it pass</li>
<li dir="auto">Refactor: improve the code while keeping tests green</li>
</ul>
<p dir="auto">The goal is fast feedback, better design (smaller units with clear contracts), and protection against regressions.<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Paradigms/Test-Driven Development.md" href="Software Engineering/06 Development Practices/Paradigms/Test-Driven Development.md" target="_blank" rel="noopener nofollow">Test-Driven Development</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What types of testing do you know?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Common categories:</p>
<ul>
<li dir="auto">Unit: test a small piece of code in isolation</li>
<li dir="auto">Integration: test how multiple components work together (DB, filesystem, HTTP, queues)</li>
<li dir="auto">Functional / End-to-end: test user-visible behavior through public APIs/UI</li>
<li dir="auto">System / Acceptance: validate the whole system against requirements</li>
<li dir="auto">Regression / Smoke: quick checks to catch breakages</li>
<li dir="auto">Non-functional: performance (load/stress), security, reliability, usability</li>
</ul>
<p dir="auto">The exact taxonomy varies by team; the key is the test scope, dependencies involved, and feedback speed.<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Paradigms/Unit Testing.md" href="Software Engineering/06 Development Practices/Paradigms/Unit Testing.md" target="_blank" rel="noopener nofollow">Unit Testing</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What are unit tests, why do we need them, and what frameworks exist?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Unit tests verify a small unit of behavior (usually a method/class) quickly and deterministically.<br>
They help prevent regressions, enable safe refactoring, document expected behavior, and improve design by forcing good boundaries.</p>
<p dir="auto">Popular .NET unit test frameworks: xUnit, NUnit, MSTest.<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Paradigms/Unit Testing.md" href="Software Engineering/06 Development Practices/Paradigms/Unit Testing.md" target="_blank" rel="noopener nofollow">Unit Testing</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What are the Arrange Act Assert blocks?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<ul>
<li dir="auto">Arrange: set up the test data and dependencies</li>
<li dir="auto">Act: execute the behavior under test</li>
<li dir="auto">Assert: verify the outcome (result, state, or interactions)</li>
</ul>
<p dir="auto">This is also known as Given When Then.<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Paradigms/Unit Testing.md" href="Software Engineering/06 Development Practices/Paradigms/Unit Testing.md" target="_blank" rel="noopener nofollow">Unit Testing</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is Moq and why is it used?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Moq is a popular .NET mocking framework used to create test doubles for interfaces (and virtual/abstract members) so you can:</p>
<ul>
<li dir="auto">isolate the unit under test from external dependencies</li>
<li dir="auto">stub return values / throw exceptions</li>
<li dir="auto">verify interactions (for example, that a dependency method was called with specific arguments)<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Paradigms/Unit Testing.md" href="Software Engineering/06 Development Practices/Paradigms/Unit Testing.md" target="_blank" rel="noopener nofollow">Unit Testing</a></em></li>
</ul>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is the difference between mocks and stubs?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">A stub provides canned answers (returns fixed data) to let the test proceed.<br>
A mock is usually used for verification: it can record calls and assert expectations about interactions (which methods were called, with what arguments, and how many times).<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Paradigms/Unit Testing.md" href="Software Engineering/06 Development Practices/Paradigms/Unit Testing.md" target="_blank" rel="noopener nofollow">Unit Testing</a></em></p>
</div></div></div><h2 id="06-development-practices-principles">Principles</h2><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is Inversion of Control (IoC)?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">IoC is a principle where the flow of control is inverted: instead of your code creating and managing its dependencies directly, an external mechanism (framework/container) coordinates object creation and composition.<br>
A common IoC technique is <a data-href="Dependency Injection" href="Dependency Injection" class="internal-link" target="_blank" rel="noopener nofollow">Dependency Injection</a>: dependencies are provided to a class (constructor/setter/parameter) rather than created inside it.<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Principles/IoC (Holywood Principle).md" href="Software Engineering/06 Development Practices/Principles/IoC (Holywood Principle).md" target="_blank" rel="noopener nofollow">IoC (Holywood Principle)</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is the Dependency Inversion Principle (DIP)?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">DIP (the "D" in SOLID) says:</p>
<ul>
<li dir="auto">High-level modules should not depend on low-level modules. Both should depend on abstractions.</li>
<li dir="auto">Abstractions should not depend on details. Details should depend on abstractions.</li>
</ul>
<p dir="auto">Practically: depend on interfaces/contracts, keep implementation details behind them, and wire concrete implementations via DI.<br>
<em><a class="internal-link" data-href="Software Engineering/06 Development Practices/Principles/IoC (Holywood Principle).md" href="Software Engineering/06 Development Practices/Principles/IoC (Holywood Principle).md" target="_blank" rel="noopener nofollow">IoC (Holywood Principle)</a></em></p>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">What is SOLID? Explain each letter.</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">SOLID is a set of design principles for maintainable OO code:</p>
<ul>
<li dir="auto">S, Single Responsibility: a module has one reason to change.</li>
<li dir="auto">O, Open Closed: open for extension, closed for modification.</li>
<li dir="auto">L, Liskov Substitution: derived types must be substitutable for their base types.</li>
<li dir="auto">I, Interface Segregation: prefer small, focused interfaces over "fat" ones.</li>
<li dir="auto">D, Dependency Inversion: depend on abstractions, not concretions.<br>

</ul>
</div></div></div><div><div data-callout-metadata="" data-callout-fold="-" data-callout="question" class="callout is-collapsible is-collapsed node-insert-event"><div class="callout-title" dir="auto"><div class="callout-icon"><svg width="16" height="16"></svg></div><div class="callout-title-inner">Which SOLID principles does Singleton violate?</div><div class="callout-fold is-collapsed"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="svg-icon lucide-chevron-down"><path d="m6 9 6 6 6-6"></path></svg></div></div><div class="callout-content" style="display: none;">
<p dir="auto">Not "by definition", but the typical Singleton usage often leads to:</p>
<ul>
<li dir="auto">DIP: code depends on a concrete global instance instead of an abstraction injected via DI.</li>
<li dir="auto">SRP: the singleton often mixes business logic with lifecycle and global access concerns.</li>
<li dir="auto">OCP: replacing/extending behavior usually requires changing call sites or the singleton itself.</li>
</ul>
<p dir="auto">A common alternative is to expose an interface and let a DI container manage a singleton lifetime.<br>

<p dir="auto">A blockchain is an append-only ledger where records are grouped into blocks and linked together using cryptographic hashes.<br>

<p dir="auto">JWT (JSON Web Token) is a compact token format for passing claims between parties.<br>
A typical JWT is <code>header.payload.signature</code> (base64url encoded) and is usually signed (JWS) so the receiver can verify integrity and issuer.<br>
Important: a signed JWT is not encrypted; anyone who has it can read the payload unless you use encryption (JWE).<br>

<p dir="auto">Cookie is a browser transport mechanism: the browser automatically sends cookies to matching origins.<br>
JWT is a token format: it can be stored and transported in cookies, or sent as <code>Authorization: Bearer &lt;token&gt;</code>.<br>
Common tradeoffs: cookies require CSRF protections (for example, <code>SameSite</code>, anti-CSRF tokens); JWTs stored in JS-accessible storage increase XSS impact.<br>
