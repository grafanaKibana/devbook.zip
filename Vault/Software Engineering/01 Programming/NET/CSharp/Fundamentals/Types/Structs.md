# Intro

A struct is a value type in C#. The variable itself holds the data â€” there is no separate heap allocation or reference indirection for locals. Assignment copies the entire value, making structs ideal for small, immutable data that represents a single logical value (coordinates, money amounts, date-time pairs). Structs implicitly derive from `System.ValueType`, are always `sealed`, and cannot participate in class inheritance.

## Deeper Explanation

```csharp
public readonly struct Money
{
    public decimal Amount { get; }
    public string Currency { get; }

    public Money(decimal amount, string currency)
    {
        Amount = amount;
        Currency = currency;
    }
}

var price = new Money(9.99m, "USD");
var copy = price;   // full bitwise copy â€” independent value
```

Key properties:

- Cannot participate in class inheritance (all structs implicitly derive from `System.ValueType`). Can implement interfaces.
- Cannot declare a finalizer. Cannot have `abstract`, `virtual`, or `protected` members.
- Always `sealed` implicitly.
- Default value is all fields zeroed (the parameterless constructor, if not explicitly defined, zero-initializes).
- Equality requires overriding `Equals`/`GetHashCode` or implementing `IEquatable<T>` â€” the default `ValueType.Equals` uses reflection and is slow.

**When to use a struct instead of a class:**

- The type logically represents a single value (like `DateTime`, `Guid`, `decimal`).
- Instance size is 16 bytes or less.
- It is immutable (or you commit to `readonly struct`).
- It is not frequently boxed.
- You need to avoid heap allocation in a hot path.

## Struct Modifiers

### readonly struct

All fields and auto-properties must be readonly. The compiler enforces immutability â€” no field or auto-property can be reassigned after construction. This prevents defensive copies when the struct is accessed through `in` parameters or `readonly` fields.

```csharp
public readonly struct Vector2
{
    public double X { get; }
    public double Y { get; }

    public Vector2(double x, double y) => (X, Y) = (x, y);

    public double Length => Math.Sqrt(X * X + Y * Y);
    public Vector2 Normalize() => new(X / Length, Y / Length);
}
```

Without `readonly`, the compiler makes a defensive copy every time a struct method is called through a `readonly` reference (e.g. an `in` parameter), because it cannot prove the method does not mutate state. With `readonly struct`, no copy is needed.

### ref struct

A `ref struct` is stack-only â€” it can never be boxed, stored on the heap, or captured by a closure/async method. This enables safe, allocation-free wrappers over stack memory.

```csharp
public ref struct SpanPair
{
    public Span<byte> First;
    public Span<byte> Second;
}
```

Key restrictions:

- Cannot be a field of a class or a non-ref struct.
- Cannot implement interfaces (would require boxing).
- Cannot be used in `async` methods, iterators, or lambdas that capture it.
- Cannot be boxed to `object`, `dynamic`, or any interface.
- `Span<T>` and `ReadOnlySpan<T>` are the most important `ref struct` types in the BCL.

C# 13 relaxes some restrictions: ref structs can now implement interfaces (with an `allows ref struct` anti-constraint) and appear in some generic contexts.

### readonly ref struct

Combines both: stack-only and fully immutable. The canonical example is `ReadOnlySpan<T>`:

```csharp
public readonly ref struct ReadOnlySpan<T>
{
    // ...internal pointer and length...
}

ReadOnlySpan<char> slice = "Hello, World!".AsSpan(0, 5);
```

This is the safest struct form â€” no heap escape, no mutation.

### partial struct

Same as partial classes â€” splits a struct definition across multiple files. The compiler merges them. Used with source generators (e.g. `System.Text.Json`, `RegexGenerator`):

```csharp
// Generated by source generator
[JsonSerializable(typeof(WeatherForecast))]
public partial struct WeatherContext { }
```

### Modifier Compatibility

| Modifier combination | Allowed? |
|---|---|
| `readonly` + `ref` | Yes (`readonly ref struct`) |
| `readonly` + `partial` | Yes |
| `ref` + `partial` | Yes |
| `abstract` | No â€” structs are implicitly sealed |
| `sealed` | Redundant â€” already implicit |
| `static` | No |

## Pitfalls

1. **Mutable structs** â€” Assigning a struct to a new variable or returning it from a property copies the value. Mutating the copy does not affect the original, which leads to silent bugs:

```csharp
struct MutablePoint { public int X; public int Y; }

var list = new List<MutablePoint> { new() { X = 1, Y = 2 } };
// list[0].X = 10;  // Compile error â€” indexer returns a copy
```

Mark structs `readonly` to make this class of bug impossible.

2. **Large struct copies** â€” A struct bigger than about 16 bytes incurs meaningful copy cost every time it is passed by value, returned, or assigned. Use `in`, `ref`, or `ref readonly` to pass large structs without copying.

3. **Default ValueType.Equals uses reflection** â€” If you do not override `Equals` and `GetHashCode`, the runtime falls back to field-by-field reflection comparison, which is extremely slow and can cause boxing for fields that are value types. Always implement `IEquatable<T>` on public structs.

4. **Boxing in generic code** â€” Calling an interface method on a struct through a non-constrained generic or through an `object`/interface variable boxes the struct onto the heap, negating the allocation benefit. Constrain generics with `where T : struct` or the interface itself to enable constrained virtual calls.

5. **Default constructor gotcha** â€” Before C# 10, structs could not have an explicit parameterless constructor. Even in C# 10+, `default(T)` and array allocation still zero-initialize without calling the constructor, so the explicit parameterless constructor is not guaranteed to run in every path.

## Questions

> [!QUESTION]- You have a high-throughput service processing 100k messages per second. Each message has an Id (Guid), a Timestamp (DateTime), and a small Status enum. Would you model this as a class or struct? Why?
> A `readonly struct` (or `readonly record struct`) is the best fit:
> - Total size is about 25 bytes (Guid 16 + DateTime 8 + enum 4) â€” slightly above the 16-byte guideline but still reasonable.
> - Value semantics mean no heap allocation per message, reducing GC pressure at 100k/s.
> - `readonly` prevents accidental mutation.
> - If the message grew to include variable-length data (strings, arrays), switching to a class would be appropriate since reference-type fields would force heap allocations anyway.

> [!QUESTION]- Why does mutating a struct returned from a property or indexer not compile (or silently do nothing)?
> Properties and indexers return a *copy* of the value, not a reference to the storage location. Assigning to a field of that copy would be discarded immediately, so the compiler blocks it (for property getters) or the change is lost (for older patterns that compile but discard the copy). This is one of the strongest arguments for making all structs `readonly` â€” it eliminates the entire category of copy-mutation bugs. Using `ref` returns or `ref readonly` returns from methods/indexers lets you hand out a reference to the actual storage if needed.

> [!QUESTION]- A colleague proposes making a 200-byte struct to avoid GC allocations. What concerns would you raise?
> Key concerns:
> - **Copy cost**: every assignment, parameter pass, and return copies 200 bytes. In a tight loop this can be worse than a single heap allocation.
> - **Stack pressure**: deep call stacks with large struct locals can cause `StackOverflowException`.
> - **Workaround complexity**: you would need `in`/`ref`/`ref readonly` everywhere, which pollutes the API surface.
> - **Better alternatives**: use a `class` with object pooling (`ObjectPool<T>` or `ArrayPool<T>`) to get both low GC pressure and reference semantics, or consider `record class` with pooling.

> [!QUESTION]- What happens when you call a non-readonly method on a struct through an `in` parameter? How does `readonly struct` fix this?
> The compiler must guarantee the `in` contract (no mutation). Since it cannot prove a non-readonly method is side-effect-free, it makes a **defensive copy** of the entire struct on every call. With `readonly struct`, the compiler knows no method can mutate state, so it skips the copy. This is why performance-sensitive structs (e.g. `Vector3`, `Matrix4x4`) should always be `readonly` â€” without it, passing them as `in` can be *slower* than passing by value due to the hidden copy on every method call.

> [!QUESTION]- You define an explicit parameterless constructor in a struct (C# 10+). Your colleague writes `MyStruct[] arr = new MyStruct[10];`. Does the constructor run? What about `default(MyStruct)`?
> Neither calls the constructor. Array allocation zero-initializes all elements without calling any constructor. `default(MyStruct)` also produces the all-zeros value, bypassing the constructor. The explicit parameterless constructor only runs when you write `new MyStruct()` explicitly. This means a struct with a parameterless constructor that sets `Id = 1` will still have `Id = 0` when obtained through `default`, arrays, or generic `new T()` with `where T : struct` (before .NET 7). This is a common source of bugs when migrating from classes to structs.

> [!QUESTION]- You need a high-performance, stack-only buffer that wraps a `Span<int>` and tracks a count. Can you use a record for this? What type would you choose?
> No â€” records cannot be `ref struct`, and `Span<T>` can only be a field in a `ref struct`. The only option is a plain `readonly ref struct`:
> ```csharp
> public readonly ref struct IntBuffer
> {
>     private readonly Span<int> _data;
>     public int Count { get; }
>     public IntBuffer(Span<int> data, int count) { _data = data; Count = count; }
>     public int this[int i] => _data[i];
> }
> ```
> You lose value-based equality, `with`, and `ToString` generation that records provide, but there is no alternative â€” `ref struct` is required, and records do not support it. If you need equality, implement `IEquatable<IntBuffer>` manually (available since C# 13 where ref structs can implement interfaces with the `allows ref struct` anti-constraint).

> [!QUESTION]- Why does `ValueType.Equals` perform so poorly on structs with reference-type fields? What should you do about it?
> The default `ValueType.Equals` uses reflection to compare fields one by one. For each field it calls `object.Equals`, which boxes value-type fields. Worse, if any field is a reference type, the runtime cannot use the fast bitwise-comparison shortcut and falls back to the slow reflective path for *all* fields. Fix: always implement `IEquatable<T>` and override `Equals`/`GetHashCode` on any public struct â€” this avoids both the reflection cost and the boxing.
